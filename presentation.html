A Vending Machine
Is an Agreement

"If you put in the right amount of money, you will get something out."

  • First implemented in hardware — gears, springs, coin slots
  • Now most vending machines run software too
  • No middleman. The machine is the agreement
  • What if we could put this pattern on a computer network?

The Ancestor of
Smart Contracts

1880s — Vending machines appear. Automated agreements, no human in the middle.
1994 — Computer scientist Nick Szabo sees the connection.
He calls the vending machine the "ancestor of smart contracts" and coins the term.
14 years before Bitcoin. Even more before Ethereum.

What Is a
Smart Contract?

Take the vending machine pattern — an agreement that enforces itself.

Remove the physical machine. Just the logic.

Put it on a network of computers that nobody owns.

Now anyone, anywhere, can use it.

An agreement that enforces itself,
running on a computer nobody controls.

February 2024 — Live on Stellar

  • Written in Rust, compiled to WebAssembly
  • Anyone can build a new machine with custom rules
  • Not just 26 buttons — unlimited possibilities
  • The old preset buttons still work too

Was Stellar the First
Smart Contract Platform?

No.

  • Ethereum (2015) — the first general-purpose smart contract platform
  • Vitalik Buterin saw Bitcoin's limitations and built a blockchain designed for programmable contracts
  • Solidity, the EVM, gas — Ethereum defined the vocabulary
  • Since then: Solana, Avalanche, Polkadot, and many more
  • Stellar chose to wait and learn — then build Soroban

Anatomy of a
Soroban Contract

A counter with one button. The simplest machine.

#[contracterror] pub enum Error { Overflow = 1 } #[contracttype] enum Key { Count } #[contract] pub struct CounterContract; #[contractimpl] impl CounterContract { pub fn increment(env: Env) -> Result<u32, Error> { let count: u32 = env.storage().instance().get(&Key::Count).unwrap_or(0); let new_count = count.checked_add(1).ok_or(Error::Overflow)?; env.storage().instance().set(&Key::Count, &new_count); Ok(new_count) } }
  • #[contracttype] — defines a symbol the contract uses as a storage key
  • #[contract] + #[contractimpl] — the machine and its buttons
  • env.storage() — the machine remembers things between presses

The Building
Blocks

Every machine is built from the same parts.

Storage
The machine's memory. Instance, persistent, or temporary.
Auth
require_auth() — verify who's pushing the button.
Events
Receipts. A log of everything that happened.
Cross-Call
Machines can call other machines.

The Soda Machine
in Code

#[contractimpl] impl SodaMachine { pub fn buy(env: Env, buyer: Address, selection: u32) { buyer.require_auth(); let price: i128 = get_price(&env, selection); // Take payment from buyer let token = token::Client::new(&env, &get_token(&env)); token.transfer(&buyer, &env.current_contract_address(), &price); // Dispense the item dispense(&env, &buyer, selection); } }
  • require_auth() — buyer proves they're really Alice
  • token.transfer() — move money from buyer to machine
  • If anything fails, everything rolls back. Atomic.

A Token Machine

A machine that creates tokens and tracks who holds them.

Token Machine
Token: LOYALTY
Total supply: 0 (so far)
Buttons: mint, transfer, burn
Mint minted 1,000 LOYALTY to Alice
Transfer Alice transferred 200 LOYALTY to Bob
Burn Bob burned 200 LOYALTY for a reward

Token Machine
in Code

#[contractimpl] impl Token { pub fn mint(env: Env, to: Address, amount: i128) { let admin = read_administrator(&env); admin.require_auth(); receive_balance(&env, to.clone(), amount); } } #[contractimpl(contracttrait)] impl TokenInterface for Token { fn transfer(env: Env, from: Address, to: Address, amount: i128) { from.require_auth(); spend_balance(&env, from.clone(), amount); receive_balance(&env, to.clone(), amount); } }
  • TokenInterface — a standard. Any machine that speaks it is a token.
  • require_auth() — only the admin can mint, only the owner can transfer

Tokens Can
Represent Anything

Same interface. Same transfer, balance, approve. Different meaning.

USDC

LOYALTY

One machine pattern.
Infinite use cases.

Atomic Swap
in Code

pub fn swap(env: Env, a: Address, b: Address, ...) { // A signs: "I'll send amount_a, I want at least min_b_for_a back" a.require_auth_for_args( (token_a, token_b, amount_a, min_b_for_a).into_val(&env), ); // B signs: "I'll send amount_b, I want at least min_a_for_b back" b.require_auth_for_args( (token_b, token_a, amount_b, min_a_for_b).into_val(&env), ); // Enforce both minimums — no one can undercut the deal if amount_b < min_b_for_a { panic!("not enough for A"); } if amount_a < min_a_for_b { panic!("not enough for B"); } // Move tokens: A→B and B→A move_token(&env, &token_a, &a, &b, amount_a, min_a_for_b); move_token(&env, &token_b, &b, &a, amount_b, min_b_for_a); }

Why Atomic
Matters

Success
Alice sends 100 USDC → Bob
Bob sends 84.75 EURC → Alice
Both sides execute. Done.
Failure
Alice sends 100 USDC → Bob
Bob's transfer fails
Everything rolls back. Alice keeps her USDC.

Both sides execute, or neither does.
Trust the machine, not the counterparty.

How Machines
Talk to Each Other

A swap machine doesn't hold tokens. It calls the token machines.

// Import another contract's interface at compile time mod token_a { soroban_sdk::contractimport!( file = "token_a.wasm" ); } // Create a client and call it let client = token_a::Client::new(&env, &contract_addr); client.transfer(&from, &to, &amount);
  • contractimport! — reads the other machine's blueprint at compile time
  • Client::new — connects to a deployed machine by address
  • Type-safe. If the interface doesn't match, it won't compile.

Beyond Finance:
What Else?

The vending machine pattern works for anything with rules.

Voting
Transparent governance. Votes are on-chain, auditable by anyone.
Supply Chain
Track provenance. Each step recorded, immutable.
Gaming
In-game items as tokens. Trade, sell, own for real.
Identity
Verifiable credentials. Prove things without revealing everything.

But Why Does
It Work?

Because we trust it.

  • Blockchains are sold as "trustless"
  • But you can't really escape trust
  • Most of us can't open a vending machine and inspect every gear, spring, and circuit
  • Most of us can't read a smart contract and know for certain it does what we expect

Sometimes the
Product Gets Stuck

You've seen it — the spiral turns, the bag tilts… and just hangs there.

Smart contracts have bugs too.

So we patch trust with auditors, evidence, and reputation.
Decentralized doesn't mean trust-free.
Trust, but verify.

The Pattern Is
Always the Same

  • A simple vending machine — money in, soda out
  • A token machine — create, transfer, burn
  • Two machines working together to swap currencies
  • A pool where people deposit assets and earn fees
  • Voting, supply chain, gaming, identity — rules enforced by code

Each one is just an agreement that enforces itself.
Open, transparent, no gatekeepers.

From Gears
to Code

1880s — Mechanical vending machines. Gears and springs enforce the deal.
1994 — Nick Szabo sees the pattern. Calls it a "smart contract."
2009 — Bitcoin. The first programmable money, but limited scripting.
2014 — Stellar launches. Fast payments, built-in DEX. 26 buttons.
2024 — Soroban goes live. Unlimited buttons. Build anything.

The same pattern Nick Szabo saw in a vending machine — now running globally, in Rust.
